# ⚡ About Me
 
I am a vibecoder. I build software at the speed of the idea — no scaffolding rituals, no six-week discovery phase, no framework debate. I find a real problem, design the system in my head, and ship it to production the same day.
 
The bottleneck in software was never typing. It was knowing what to build, deciding what to leave out, and having the taste to tell a good architecture from a plausible one. AI collapsed the cost of the first draft to near zero — so the whole game moved to judgement, and that is where I spend my attention.
 
**How that plays out in practice:**
 
- **Ship first, refactor when it earns it.** Every project here started because I personally needed it. If nobody uses it, it was never worth architecting properly.
- **Single-file architecture by default.** One HTML file, CDN imports, localStorage, deployed to Vercel. No build step, no dependency graph, no bundler config. Boring to deploy, impossible to break.
- **AI as leverage, not autopilot.** I direct the build, own the architecture, and review every line that matters — schema design, auth boundaries, state, security. AI writes the parts that were always going to be boilerplate.
- **Production or it doesn't count.** Not tutorials, not clones, not `todo-app-v3`. Real users, real data, real deployments, real bugs at 2am.
**Volume is the strategy.** Ten shipped products teach you more than one perfect one. Each build below was scoped in a session, shipped in a session, and then survived contact with actual use.

# 🧬 How I Build
 
```yaml
Loop:
  1. Feel a problem personally
  2. Scope it in one conversation
  3. Build it in one session
  4. Deploy before losing interest
  5. Use it daily, or delete it
 
Defaults:
  Architecture:   single-file HTML or React via CDN
  Persistence:    localStorage first, Supabase when sync matters
  Auth:           Supabase GoTrue + row-level security
  AI:             Anthropic API behind a Vercel serverless proxy
  Deploy:         commit to GitHub, live in 30 seconds
  Secrets:        environment variables, never the client
 
Non-Negotiable:
  - No API key ever reaches the browser
  - Row-level security from the first table, not bolted on later
  - Backup and restore in v1, not v2
  - If deployment takes more than 30 seconds, it is wrong
```

# 🚀 Shipped
 
<details>
<summary><b>🧠 Atlas — Local-First Personal AI Operating System</b></summary>
<br/>
A local-first Python orchestrator that routes natural-language requests to domain specialists, remembers context across sessions and responds through a voice-driven browser HUD.
 
| Category | Details |
|---|---|
| Stack | Python, SQLite, TF-IDF Retrieval, Anthropic API, Web Speech API |
| Scale | Four specialist agents behind a routing layer with persistent cross-session memory |
| Performance | Local-first execution with no retrieval round-trip; sub-second wake-word activation |
| Security | Fully local memory persistence, no third-party data residency, secrets isolated from repository |
| Impact | Replaces four separate tools with one voice-addressable surface |
| Repository | Available on GitHub |
 
Intent classification is separated from execution: a routing layer inspects each request, selects the appropriate specialist and hands off with a structured contract. Memory is a SQLite store queried by TF-IDF cosine similarity, improving recall as the corpus grows without any vector-database dependency.
 
</details>
<br/>
<details>
<summary><b>🚆 RIVO — Voice-First UK Rail Assistant</b></summary>
<br/>
A conversational rail assistant answering live departure, arrival and accessibility queries by voice, designed for the moment you are walking to a platform and cannot look at a screen.
 
| Category | Details |
|---|---|
| Stack | FastAPI, Python, RapidFuzz, Web Speech API, PWA, Vercel |
| Scale | 2,623-station resolver, multi-turn conversational context, full Passenger Assist flow |
| Performance | Millisecond fuzzy station resolution, timezone-correct via zoneinfo with DST handling |
| Security | Credentials held server-side in environment configuration, no client-exposed tokens |
| Impact | Accessibility-first design with Passenger Assist and a live position estimator |
| Repository | Available on GitHub |
 
The hard problem is not the API call but resolving what a person actually said into a station code. Conversational state carries across turns, so follow-up questions resolve without restating origin and destination.
 
</details>
<br/>
<details>
<summary><b>🍳 ImSoCooked — Student Recipe & Grocery Budget Platform</b></summary>
<br/>
A React Native application connecting what students cook to what it actually costs, with live grocery pricing and behavioural insight generation.
 
| Category | Details |
|---|---|
| Stack | React Native, Expo, TypeScript, Supabase, PostgreSQL, Pricing API |
| Scale | Live price resolution across grocery baskets with an append-only event store |
| Performance | Chunked API resolution tuned to optimal batch size, high reliable match rate under guard |
| Security | Supabase row-level security, verified-match-only write pattern, strict secret hygiene |
| Impact | Turns recipe browsing into an accurate weekly grocery budget |
| Repository | Available on GitHub |
 
External pricing APIs return noisy matches, so ingestion uses an allowlist-and-guard pattern: only verified matches are written. An append-only event table underpins the insights layer, enabling time-based pattern analysis without mutating history.
 
</details>
<br/>
<details>
<summary><b>💰 Spendwise — Personal Finance Intelligence Platform</b></summary>
<br/>
A finance platform built against real bank data, with categorisation, subscription detection, group splitting and UK tax-year reporting.
 
| Category | Details |
|---|---|
| Stack | JavaScript, PWA, Local Persistence, Vercel, PDF Export Pipeline |
| Scale | 200+ UK merchant categorisation rules over a live transaction corpus |
| Performance | Offline-first architecture with instant global search across every transaction |
| Security | Client-side persistence only, no financial data leaves the device |
| Impact | Subscription Manager, Group Splits, UK Tax Year reporting, receipt attachments |
| Repository | Available on GitHub |
 
Categorisation is the core engineering problem: bank descriptors are inconsistent and truncated. A rules engine handles the known patterns, with a duplicate-review modal keeping a human in the loop exactly where automation is unreliable.
 
</details>

A production trading platform delivered commercially for the Inter-European Asset Management Competition.
 
| Category | Details |

| Stack | Full-Stack Web, Supabase, Real-Time Subscriptions |
| Scale | ~150 concurrent traders across 16 competing teams |
| Performance | Real-time position and leaderboard propagation under concurrent load |
| Security | Team-scoped access control with an auditable transaction ledger |
| Impact | Commercially delivered, sponsor-funded engagement |

A genuine multi-tenant concurrency problem: 16 teams trading simultaneously against shared market state with ledger correctness as a hard requirement. Vibecoding got it demoed in days; the ledger got the slow, careful treatment it deserved.

<summary><b>📅 MyDay — Productivity & Academic Operating System</b></summary>
<br/>
A cross-device productivity platform integrating calendar, tasks, habits, focus sessions and university timetabling into one installable surface.
 
| Category | Details |
|---|---|
| Stack | PWA, Supabase, Vercel Serverless, Capacitor iOS, iCal |
| Scale | Six integrated modules with cross-device synchronisation |
| Performance | Serverless iCal proxy with automated timetable change-detection |
| Security | Supabase row-level security enforcing per-user data isolation |
| Impact | Installed as a daily-driver PWA with a native iOS wrapper |
| Repository | Available on GitHub |
 
The interesting constraint was the university timetable feed: an iCal endpoint unreachable from the browser that changes without notice. A serverless proxy runs change-detection against the previous snapshot, surfacing changes as notifications rather than as a missed lecture.
 
</details>
